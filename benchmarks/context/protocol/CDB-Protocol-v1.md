# Context-Dependence Benchmark ‚ÄĒ Protocol v1 (CDB-v1)

> **FROZEN once the first scored record is published.** No section, gold rule, grading
> rule, hypothesis, or fairness guard below may change after that point except via a dated
> entry in the ExecPlan Decision log (`context-dependence-benchmark-scorecrux-2026-07-03`).
> Improvements ship as **CDB-v2** with a new protocol, never in-place edits.

## 1. What CDB measures

For **any agent stack** and **any memory/context backend**, how much a task depends on
*carried context*, and how well a backend supplies it ‚ÄĒ broken into **context-driven
sections** by knowledge type. The result is a **fingerprint across sections**, not a single
number. A backend that scores high everywhere including the negative controls is *leaking*,
not good.

## 2. The single variable

Per cell `(section, backend, seed)`: a cold agent (fixed model, fixed turn cap, **no live
tool calls**) answers the section's probes. The **only** thing that varies between backends
is the **context block the backend supplies** (its `assemble()` output), prepended to an
otherwise identical prompt. This isolates *context supply* from *tool-reach* and is fair to
backends that have no MCP surface. Parity is enforced: prompts differ only by the injected
block (guarded by diff).

## 3. Backends (arms)

A backend implements `plant(prior_knowledge)` and `assemble(probes, token_budget) -> block`.
Reference backends shipped:

- `none` ‚ÄĒ empty block (the floor).
- `vendor-native` ‚ÄĒ the honest competitor: a rules-file dump (CLAUDE.md-style) of the prior
  knowledge, **including stale entries where present** (simulates the naive persistent file).
- `crux` ‚ÄĒ the Crux daemon: facts planted via `/v1/facts`, `assemble` returns the
  **current** (freshness-resolved) slice rendered as the `## Crux Context` bundle.
- `oracle` *(calibration)* ‚ÄĒ returns the gold verbatim. Must ceiling every section.
- `random` *(calibration)* ‚ÄĒ returns junk. Must floor every section.

Community backends (`mem0`, `zep`, `letta`, ‚Ä¶) are PRs implementing the same interface.

## 4. Sections and pre-registered hypotheses

| # | Section | Knowledge type | Gold | Hypothesis |
|---|---|---|---|---|
| **S1** | rederivable *(CONTROL)* | value is present in a sandbox file the agent can read | that value | **~0 lift** for all backends; a real backend must not beat `none` here. Lift ‚áí **leak flag** |
| **S2** | arbitrary | non-rederivable codename/port/id, only in prior knowledge | that value | high lift over `none`; `none` ‚Čą 0 |
| **S3** | cross-session | a decision from a prior session governs the probe | decided value | high lift; `none` ‚Čą 0 |
| **S4** | causal/why | rationale ("why X over Y") only in prior knowledge | the reason token | high lift; `none` ‚Čą 0 |
| **S5** | supersession *(CONTROL)* | a fact was set then **changed**; only the current value is correct | the **current** value | **naive memory fails** (surfaces stale); only freshness-aware backends win |

S6 (scale/needle) and S7 (coordination/multi-agent) are **CDB-v1.1** ‚ÄĒ scaffolded, not in the
v1 scored set (S6 needs the 100k‚Äď2M-token corpus; S7 needs the multi-agent scenario harness).

## 5. Grading (deterministic)

Each probe carries a `must_contain` regex over the **current** gold. A probe is correct iff
the agent's `answers.json[probe]` is non-empty, not `UNKNOWN`, and matches the regex
(case-insensitive). No LLM judge in v1. `recall = correct / n_probes`. Per cell we also record
`delta_vs_none` and `delta_vs_vendor_native`, McNemar cells vs `none`, token cost (USD), and
the safety gate (`s_gate=0` ‚áí record scores 0 Em regardless).

## 6. Fairness guards (all mandatory)

- **Determinism:** gold generated by `random.Random(hash(section, seed))`; same seed ‚áí
  byte-identical gold. Instances are fresh per seed so no backend can overfit.
- **Parity:** on/none prompts differ only by the injected block (diff-guarded).
- **Counterbalancing:** backend run order rotated per seed.
- **No-leak invariant:** on S1, every backend's lift over `none` must be within CI of 0.
  A `crux` (or any real backend) lift on S1 **fails the build**.
- **Calibration:** `oracle` must ceiling and `random` must floor every section, or the scale
  is broken.
- **Corpus + model identity** on every record (`corpus`, `model`); judge lineage N/A (no judge).
- **Report either way:** every cell published win or lose; raw transcripts + manifest hashes
  retained.

## 7. Effective Minutes mapping

Sections feed ScoreCrux Em continuity dimensions: S2/S3 ‚Üí `K_decision`, S4 ‚Üí `K_causal`,
S5 ‚Üí `K_checkpoint`. `cx_em` per cell = recall-weighted, safety-gated, per the METRICS spec.

## 8. Integrity

Each cell emits a manifest `{suite_version, section, backend, model, seed, corpus,
prompt_sha256, gold_sha256, answers_sha256, scores}`. Published seeds regenerate identical
gold; a third party reproduces a row on a clean machine with no CueCrux software (the crux
backend is optional ‚ÄĒ the `none`/`vendor-native`/`oracle`/`random` arms need nothing).

  `plant()/assemble(task, token_budget)` contract so every backend is on equal footing.
- **2026-07-03:** Published records now carry `gold_sha256` (the canonical
  `sha256(gen_case(section, seed))`) so a third party can run `verify_manifest.py` directly on a
  `public-data/context/*.json` file ‚ÄĒ previously the field lived only in the per-cell
  `runs/.../manifest.json`. Integrity semantics per ¬ß8 are unchanged.
- **2026-07-03:** The `crux` adapter's default `CRUX_BASE` is `http://127.0.0.1:14800` (was a
  hardcoded tailnet IP). The daemon endpoint is operator-supplied via env; this does not affect
  any non-`crux` arm, none of which touch the network.
